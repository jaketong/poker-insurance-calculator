# REMOTE_AI_DEV_TOOL_STACK.md

## 1. 文件目的

本文件用于**审计和规划**远程 AI 开发系统**可选**的工具与接入顺序。

- **不是**安装教程，也**不代表**当前要立即接入任何一项。  
- 作用是把「ChatGPT + Cursor + Codex + GitHub + Netlify + 腾讯云服务器」之外的补充工具，按**职责、风险、建议**写清楚，避免后续无序扩张。  
- 与 `LOBSTER_SYSTEM_RULES_AUDIT.md` 配合：审计表管**规则与流程**，本文件管**工具栈认知与优先级**。

---

## 2. 当前阶段原则（V1.1）

当前 V1.1 阶段遵循：

1. **先规则** — 边界、高风险项、谁负责什么，先写进文档与对话约定。  
2. **再流程** — 任务下发、执行、审查、提交、部署、验收、复盘，顺序固定。  
3. **再工具** — 在流程跑稳的前提下，再评估 CLI、CI、监控等是否值得接。  
4. **再服务器** — 服务器与远程控制增强，放在主线稳定之后。  
5. **再自动化** — n8n、Bot、多 Agent 等编排，最后才考虑。

**本次范围**：仅新增本工具栈审计文档；**不**安装、**不**配置、**不**接任何 API、**不**改现有部署流程。

## 4. 可补充工具总表

| 工具 | 主要作用 | 能减少什么低价值操作 | 是否涉及密钥或权限 | 是否可能影响部署 | 当前建议 |
|------|----------|----------------------|--------------------|-------------------|----------|
| GitHub CLI `gh` | 终端查 PR、Release、状态，减少网页点点点 | 查 issue/PR、看检查状态 | 是（需登录 Token） | 间接（误操作可合并/发版） | 优先**评估**，不马上装 |
| GitHub Actions | CI：lint、test、build 自动化 | 本地忘跑检查、回归漏测 | 是（`GITHUB_TOKEN`、Secrets） | 是（可阻断合并或触发发布） | 优先**评估**；若上，从**最小** check 开始 |
| Netlify API / CLI | 查站点、部署列表、日志 | 打开控制台查 Published | 是（Personal Access Token） | 是（若误用可触发/改配置） | 后期**只读**查状态；不用于随意触发生产 |
| Tailscale | 安全组网，远程 SSH/RDP | 公网暴露、手工配 VPN | 是（设备与 ACL） | 否（默认不替代 Netlify） | 服务器阶段**前**评估 |
| Playwright | 浏览器 E2E、截图对比 | 手工点主要路径验收 | 一般低（测试环境为主） | 否（与生产发布解耦时） | **后期**线上自动验收 |
| UptimeRobot | HTTP 存活与延迟监控 | 人工发现站点挂了 | 是（监控 URL、告警渠道） | 否 | **后期**可访问性监控 |
| Sentry | 前端错误与性能监控 | 用户截图「白屏」无法复现 | 是（DSN / 项目密钥） | 否（SDK 接入方式得当） | **后期**错误监控 |
| Telegram / 飞书 / Discord Bot | 部署结果、审查提醒、确认流 | 盯邮箱和 Netlify 列表 | 是（Bot Token、Webhook） | 间接（若绑部署钩子） | **后期**手机通知与确认提醒 |
| n8n | 流程编排、Webhook 串联 | 人肉串联多系统 | 是（凭据、Webhook） | 是（可触发外部动作） | **中后期**；当前**不接** |
| Claude Code / Devin / Codegen 等 AI Coding Agent | 另一套自动写码路径 | Cursor 重复劳动 | 视产品（多密钥、多上下文） | 是（若直连仓库） | **暂不优先**，避免多 Agent 混乱 |

## 5. 工具接入优先级（建议顺序）

以下为**规划顺序**，不是当前执行清单：

1. **GitHub CLI `gh`**：优先评估是否值得装；**不**为本文档立刻安装。  
2. **GitHub Actions**：优先评估；若落地，**只**从最小 `build` / `test` / `lint` 检查开始。  
3. **Netlify API / CLI**：后续**仅**用于查询部署状态、日志；**不**用于随意触发生产部署。  
4. **Tailscale**：进入服务器强化阶段**前**评估，用于更安全的远程连接。  
5. **Playwright**：后期用于线上或预发环境的自动验收。  
6. **UptimeRobot**：后期用于网站可访问性监控。  
7. **Sentry**：后期用于前端错误监控。  
8. **Telegram / 飞书 / Discord Bot**：后期用于通知与「人点确认」类提醒。  
9. **n8n**：中后期流程编排；**当前**不接入。  
10. **Claude Code / Devin / Codegen**：不优先，避免与 Cursor 主线并行混乱。

## 7. 高风险停止规则

未来若评估或接入任一工具，出现以下情况须**停止并向用户确认**：

- 需要新增密钥 / Token（含只读 Token 若策略要求）  
- 需要修改 `package.json`  
- 需要新增依赖  
- 需要修改 GitHub Actions（或等效 CI 定义）  
- 需要修改 Netlify 配置（Build、Publish、分支、环境变量、域名等）  
- 需要触发生产部署（非用户明确授权的自动路径）  
- 需要自动 `git push`  
- 需要服务器开放新端口或改安全组  
- 需要改 `.env` 或把密钥写入仓库  
- 需要多个 Agent **同时**改同一套代码库且无总控  
- 需要删除文件（非任务明示）  
- 需要修改扑克保险**核心算法**或**赔率表**（与工具无关但属项目红线）

## 8. 后续接入顺序建议

1. 先完成并稳定**工具栈文档**（即本文件与规则审计表）。  
2. 再**单独**评估 GitHub CLI `gh`（是否真省时间、团队是否愿意维护 Token）。  
3. 再设计**最小** GitHub Actions 检查（例如仅 `npm run build`，不扩大）。  
4. 再评估 **Netlify 状态查询**（只读 API 或人工流程是否足够）。  
5. 再进入**服务器安全连接**评估（Tailscale 或其它方案对比）。  
6. **最后**才考虑 n8n、通知 Bot、多 Agent 与重度自动化编排。

---

## 9. 当前结论

`REMOTE_AI_DEV_TOOL_STACK.md` **仅为**工具栈审计与规划文档，**不涉及**代码、算法、依赖、部署配置文件或服务器上的实际配置变更。

当前阶段用途：

- 统一团队对「还有哪些工具、各自干什么、风险在哪」的认知；  
- 固定**接入顺序**，避免乱装工具、乱接自动化、浪费额度与扩大风险面。
