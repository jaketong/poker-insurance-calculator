# GITHUB_CLI_INSTALL_EVALUATION.md

## 1. 文件目的

本文件用于**单独评估** **GitHub CLI（`gh`）** 是否值得安装；若未来安装，应装在**何处**、解决**什么问题**、存在**哪些风险**，以及如何**最小化、安全地**接入。

**明确声明：**

- 本文件**不是**安装教程，**不代表**当前要立即安装 `gh`。  
- **不**执行任何 `gh` 命令，**不**进行 `gh auth login`，**不**配置 Token。  
- 仅作为 **V1.1「远程开发流程协议化」** 下的**决策依据**与后续立项参考。

---

## 2. 当前阶段原则（V1.1）

1. **先规则** — 角色、边界、高风险项先固定。  
2. **再流程** — 任务、验收、提交、push、部署检查顺序跑顺。  
3. **再工具** — 在流程稳定后再评估 CLI 等工具是否「值得」。  
4. **再服务器** — 服务器侧能力放在主线之后。  
5. **再自动化** — 编排、脚本化、多系统集成最后考虑。

**本次范围：** 只评估 **`gh` 是否值得安装**；**不**安装、**不**登录、**不**配置、**不**接入。

## 4. `gh` 可能解决的问题

| 场景 | 当前手动做法 | `gh` 可能提供的帮助 | 是否现在刚需 | 备注 |
|------|----------------|---------------------|--------------|------|
| 1. 查看仓库状态 | 打开 GitHub 仓库页看分支、最近活动 | `repo view` 等只读信息聚合 | **否** | 低频时可网页替代 |
| 2. 查看最近 commit | 网页 Commits 列表或本地 `git log` | `api` / `browse` 跳转、列表过滤 | **否** | 本地已有 `git log` |
| 3. 查看分支 | 网页 Branches 或本地 `git branch -a` | 远端分支列表、保护规则提示等 | **否** | 视习惯而定 |
| 4. 查看 Pull Request | 网页 PR 列表与 diff | `pr list` / `pr view` | **否** | 单人或小团队可能 PR 少 |
| 5. 查看 GitHub Actions 运行结果 | 网页 Actions 标签 | `run list` / `run view` | **否** | 当前未强制 CI |
| 6. 未来在服务器上查询 GitHub 状态 | SSH 上去再开浏览器或 curl | 终端只读查询（若已登录） | **否** | 涉及凭据落点，高风险 |
| 7. 未来配合脚本做发布状态汇总 | 复制粘贴网页信息 | JSON 输出便于脚本拼接 | **否** | 与自动化阶段绑定 |
| 8. 减少反复打开 GitHub 网页 | 多次切换标签查 PR/Actions | 终端一站式只读查询 | **否** | 频次够高才值得 |

## 5. 安装位置评估

| 安装位置 | 优点 | 缺点 | 风险 | 当前建议 |
|----------|------|------|------|----------|
| 1. 本地电脑 | 与开发者身份绑定清晰；易卸载；不扩大服务器攻击面 | 每人一套环境，需各自维护登录 | Token 存本机仍属敏感，需防钓鱼与备份盘泄露 | **若未来安装，优先评估本机** |
| 2. 腾讯云服务器 | 远程会话内一站式查 GitHub，少开浏览器 | 长期驻留、多用户共享、备份镜像含凭据 | **凭据在服务器面暴露面更大**；合规与轮换更复杂 | **当前不建议**也**不建议马上**装到服务器；须**单独立项** |
| 3. 暂不安装，只保留网页检查 | 零新增凭据面；流程已够用 | 偶发多点几次网页 | 低 | **当前默认推荐** |

**结论（必须写明）：**  
**当前不建议马上安装到腾讯云服务器。** 若未来安装，**优先评估本地电脑**或**临时、只读、短周期**查询用途。服务器安装涉及**长期凭据、Token、权限与安全边界**，必须**单独立项**并由用户确认。

---

## 6. Token 和权限风险

`gh` 的多数有价值能力依赖 **GitHub 登录** 与 **Token / 凭据缓存**，属于**敏感权限面**。

必须遵守：

1. **Token / 登录凭据**属于敏感权限，按公司/个人安全策略管理。  
2. **不得**把 Token 写入本仓库文档或聊天可复制明文（原则：最小暴露、用完即撤）。  
3. **不得**把 Token 写入 `.env` 并 **commit** 进 Git。  
4. **不得**把 Token **发给** Cursor / Codex 或贴进可被日志长期保存的上下文。  
5. **不得**让 AI **自行生成、代管或长期保存** Token。  
6. 未来若确需 Token，必须**单独确认**权限范围（只读 / PR / repo 等）与轮换策略。  
7. **服务器保存 Token 风险更高**（多进程、备份、共享账户、入侵后果更大）。

## 7. 不安装 `gh` 的替代方案

当前**不安装** `gh`，仍可完整走通：

1. **GitHub 网页** — 仓库、Commits、PR、Actions。  
2. **Netlify Deploys** — 部署状态与日志。  
3. **Cursor 返回** `git status` / `git log` / `git diff` — 本地与待推送状态。  
4. **ChatGPT** 验收 — 对照任务书与规则文档。  
5. **Codex** 审查 — 必要时。

**结论（必须写明）：**  
**当前不安装 `gh` 不会阻塞 V1.1 流程。**

---

## 8. 当前是否值得安装

**当前结论：**

- **`gh` 有价值，但不是现在的刚需。**  
- **当前不建议立即安装。**  
- 建议**继续**使用 **GitHub 网页 + Cursor 内建 `git` 命令 + Netlify 页面** 完成现有流程。  
- 当出现以下信号之一，再**单独立项**评估安装：  
  - 需要**频繁**查询远端 PR / Actions / Release 状态；  
  - 进入**服务器控制中心**阶段且明确需要终端侧只读查询 GitHub。

## 9. 未来最小安全安装前提

若未来要安装 `gh`，**至少**先满足（书面或对话留痕）：

1. **明确安装位置**（本机 / 专用跳板机 / 服务器——后者须额外安全评审）。  
2. **明确使用目的**（只读查询 / PR 管理 / release 等，禁止范围外操作）。  
3. **明确是否需要登录**（是/否；若否，是否仅用公开 API 的有限能力）。  
4. **明确 Token 权限**（最小权限原则）。  
5. **明确 Token 保存位置**（系统凭据管理器、专用 vault，禁止进仓库）。  
6. **明确谁可以使用**（个人 / 角色 / 是否共享账号）。  
7. **明确不得自动 `git push`**（与 `GITHUB_NETLIFY_RELEASE_FLOW.md` 一致）。  
8. **明确不得自动部署**（含 Netlify hook 等）。  
9. **明确不得绕过用户确认**（发布链路不变）。  
10. **明确失败时停止规则**（异常即停、不扩大变更）。

---

## 10. 高风险停止规则

未来在**评估或安装** `gh` 时，出现以下任一情况须**停止**并向用户确认：

1. 需要 **GitHub Token** 且用途或范围不清。  
2. 需要 **`gh auth login`** 但未定义凭据归属与轮换。  
3. 需要把凭据**长期保存到服务器**且无独立安全评审。  
4. 需要配置 **GitHub Secrets**（与「仅装 CLI」常无关，但若脚本化则易出现）。  
5. 需要修改 **`.env`** 以存放 Token 并存在误提交风险。  
6. 需要修改 **Git 配置**（user、remote、credential helper 等）以「配合 gh」。  
7. 需要修改**远程地址**或引入镜像。  
8. 需要 **自动 push**。  
9. 需要 **自动部署** 生产。  
10. 需要修改 **Netlify** 配置。  
11. 需要修改 **GitHub Actions** / `.github`。  
12. 需要**服务器权限**或开放新端口且无变更说明。  
13. **无法判断** Token 权限范围是否过大。  
14. **用户未明确确认**安装与使用范围。  

---

## 11. 后续建议

1. **先**完成并归档本评估文档。  
2. **不**立即安装 `gh`。  
3. **继续**用现有流程完成 **1–2** 个真实任务，观察是否真的被「反复查 GitHub 网页」卡住。  
4. 若频次变高，再**单独立项**：**仅评估本地安装 `gh` + 只读使用范围**。  
5. 若进入**服务器控制中心**阶段，再**单独**评估「服务器是否装 `gh`」及凭据方案。  
6. **任何**安装、登录、Token、服务器凭据，都必须**用户单独确认**并更新相关规则文档。

---

## 12. 当前结论

- **GitHub CLI `gh`** 对未来远程 AI 开发流程有**潜在价值**，但**当前不是必须工具**。  
- **当前阶段：不安装、不登录、不执行 `gh` 命令、不配置 Token。**  
- 本文件**只**作为后续决策依据，**不改变**现有 **GitHub / Netlify** 发布流程与「用户确认 push」的纪律。
