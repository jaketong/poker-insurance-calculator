# GITHUB_CLI_ACTIONS_MINIMAL_EVALUATION.md

## 1. 文件目的

本文件用于评估 **GitHub CLI（`gh`）** 与 **GitHub Actions** 在「龙虾式远程 AI 开发系统」中的**最小接入价值**、**风险**与**后续接入顺序**。

**明确声明：**

- 本文件**不是**安装教程，**不代表**当前要立即接入任一项。  
- **不**创建 workflow，**不**改变现有 **GitHub → Netlify** 发布链路。  
- 仅做**纸面评估**，为后续「单独立项、单独确认」提供依据。

---

## 2. 当前阶段原则（V1.1）

当前阶段遵循：

1. **先规则** — 边界、角色、高风险项先写清。  
2. **再流程** — 任务、验收、提交、push、部署检查顺序固定。  
3. **再工具** — 在流程稳定后再评估 CLI、CI 等是否值得接。  
4. **再服务器** — 服务器与远程控制增强，放在主线之后。  
5. **再自动化** — 编排、Bot、多 Agent 等最后考虑。

**本次范围：** 只评估；**不**安装、**不**配置、**不**创建 GitHub Actions、**不**修改 GitHub / Netlify / 腾讯云上的既有配置。

---

## 3. 当前已有发布流程基础

### 3.1 已完成的流程与规划文档（摘录）

1. `LOBSTER_SYSTEM_RULES_AUDIT.md` — 总控规则审计。  
2. `REMOTE_AI_DEV_TOOL_STACK.md` — 工具栈与接入优先级。  
3. `GITHUB_NETLIFY_RELEASE_FLOW.md` — GitHub / Netlify 发布与验收流程。

### 3.2 当前已跑通的人机协作链路

1. **ChatGPT** 总控（目标、风险、验收）。  
2. **Cursor** 执行（改代码、跑检查、回传结果）。  
3. **Codex** 审查（越界与质量）。  
4. **GitHub** 保存版本（commit 历史）。  
5. **Netlify** 在 push 后**自动部署**（以项目当前连接为准）。  
6. **用户**明确确认 **push**。  
7. **用户**检查 Netlify **Published** 与 **commit 一致**。  
8. **用户**做**线上人工验收**。

在此之上，再讨论 `gh` / Actions 是否「锦上添花」，而不是替代上述纪律。

## 4. GitHub CLI `gh` 的作用评估

| 能力 | 能减少的低价值操作 | 是否需要登录或 Token | 当前是否建议接入 | 风险说明 |
|------|----------------------|------------------------|------------------|----------|
| 1. 查看仓库状态 | 少开网页、少点多层菜单 | 多数查询需已登录或配置凭据 | **仅评估，不接** | Token 管理、误用生产账号 |
| 2. 查看 PR | 快速列 PR、看 diff 摘要 | 通常需要 | **仅评估，不接** | 误 merge / 误关 PR 若权限过大 |
| 3. 查看 GitHub Actions 运行状态 | 终端看 workflow 结论 | 需要 | **仅评估，不接** | 与「尚未建 workflow」阶段关系弱 |
| 4. 查看 commit / branch | 对照本地与远端、看 tag | 读操作常需登录 | **仅评估，不接** | 低；但仍属凭据面 |
| 5. 未来配合脚本查询发布状态 | 可与检查清单拼接半自动 | 需要 Token / 应用身份 | **仅评估，不接** | 脚本化后易「顺手加功能」扩大面 |
| 6. 未来辅助远程服务器查询 GitHub 信息 | SSH 会话里少切换浏览器 | 需要安全存放 Token | **仅评估，不接** | 服务器侧密钥与 ACL 风险 |

**当前结论（必须遵守）：**  
**GitHub CLI `gh` 可以优先评估**，但**当前不安装、不登录、不接入**任何 `gh` 命令路径。

## 5. GitHub Actions 的作用评估

| 能力 | 能减少的低价值操作 | 是否会影响发布 | 当前是否建议接入 | 风险说明 |
|------|----------------------|----------------|------------------|----------|
| 1. 自动运行 `build` | 减少「本地忘了 build 就 push」 | 不直接部署 Netlify，但可阻断合并 | **仅评估，不接** | workflow 写错会误伤团队节奏 |
| 2. 自动运行 `test` | 回归更稳定（若项目已有 test 脚本） | 同上 | **仅评估，不接** | 无测试却硬上会噪音 |
| 3. 检查是否误改 package / lock | PR 或 push 前机械拦截 | 可阻断合并 | **仅评估，不接** | 规则过严会误杀合法变更 |
| 4. 检查是否误改 src / 配置 | 路径级守卫 | 可阻断合并 | **仅评估，不接** | 需维护允许列表，否则摩擦大 |
| 5. 检查 Markdown 文档格式 | 统一排版、少人工挑刺 | 一般不影响 Netlify | **仅评估，不接** | 选错工具链会拖慢 CI |
| 6. 后续配合 PR 做预检查 | 把风险拦在合并前 | 依策略可能「无绿不过」 | **仅评估，不接** | 与分支保护、权限模型绑定 |

**当前结论（必须遵守）：**  
**GitHub Actions 后续可以评估最小接入**，但**当前不创建 workflow**，**不**修改 `.github/workflows`（含不新建该目录下的任何文件）。

## 6. 最小 GitHub Actions 设想（仅设想）

未来若单独立项通过，**最小** workflow 可以从下列步骤**设想**（**本文不创建任何文件**）：

1. `checkout` 仓库代码；  
2. 安装依赖（与当前 lock 策略一致，避免隐式升级）；  
3. 运行 `npm run build`；  
4. **若**仓库已存在约定好的 `npm test`，再运行 `npm test`；  
5. 在日志中明确输出成功 / 失败原因。

**现在不创建 workflow 的原因：**  
创建 `.github/workflows` 属于**流程变更**，必须**单独立项、单独确认**，并与 `GITHUB_NETLIFY_RELEASE_FLOW.md` 中的发布纪律对齐。

---

## 7. 与 Netlify 的关系

1. **GitHub Actions** 是**检查器**（质量闸门、自动化脚本）。  
2. **Netlify** 是**部署器**（构建与托管站点）。  
3. **当前生产部署**仍由 **push 到 `master`** 后触发 **Netlify 自动部署**（以现有项目配置为准）。  
4. **Actions 不应替代 Netlify** 承担站点托管职责。  
5. **Actions 不应自动触发生产部署**（例如调用 Netlify build hook 需谨慎，默认不做）。  
6. **Actions 不应修改 Netlify 配置**（API 改站点设置属高风险）。

**明确：** 未来即使接入 GitHub Actions，也必须**避免改变**现有 **Netlify 自动部署规则**与「用户确认 push → 检查 Published」的主线，除非用户书面/对话**单独**批准整体变更方案。

## 9. 高风险停止规则

未来在**评估或接入** `gh` / Actions 过程中，如出现以下任一情况，须**停止**并向用户确认：

1. 需要 **GitHub Token**（个人或 PAT）且用途不清。  
2. 需要 **GitHub Secrets** 才能继续。  
3. 需要修改已有 **`.github/workflows`**。  
4. 需要**新增** workflow 文件。  
5. 需要修改 **`package.json`**。  
6. 需要修改 **lock** 文件。  
7. 需要**新增依赖**才能完成 CI。  
8. 需要修改 **Netlify** 配置。  
9. 需要**触发生产部署**（含误触 hook）。  
10. 需要 **自动 `git push`**。  
11. 需要**服务器权限**或开放新端口。  
12. 需要修改 **`.env`** 或把密钥写入仓库。  
13. 需要**多个 Agent** 同时改同一仓库且无总控。  
14. **无法判断**是否会影响现有 **发布流程** 或 **Netlify 行为**。

## 10. 后续最小接入顺序建议

1. **先**完成并归档本评估文档（你正在读的这一版）。  
2. **再**单独开任务：「是否值得在本机安装 `gh`」——产出**是/否 + 原因 + 替代方案**（网页是否足够）。  
3. 若评估为「值得」，**只做**只读类 `gh` 能力验证（仍须用户确认命令范围与凭据存放方式）。  
4. **再**单独设计 **GitHub Actions 最小 workflow**（写在设计文档里，仍可不落地）。  
5. **优先**在分支或 **PR** 场景验证 workflow，不直接绑死 `master`。  
6. 确认 **不会干扰 Netlify** 的触发逻辑与预期后，再讨论是否纳入 `master` 流程。  
7. **任何**改变发布链路的行为，都必须**用户单独确认**并回写 `GITHUB_NETLIFY_RELEASE_FLOW.md` 等文档。

---

## 11. 当前结论

- **GitHub CLI `gh`** 与 **GitHub Actions** 对远程 AI 开发流程有**潜在价值**（省重复点击、提高 build/test 纪律），但当前项目**只进入评估阶段**。  
- **当前不安装、不配置、不创建 workflow、不改变** GitHub / Netlify 既有发布链路。  
- **下一步**若继续：只能先进入 **「`gh` 是否值得安装」的单独评估**，**不得**在未立项的情况下直接安装或登录。
